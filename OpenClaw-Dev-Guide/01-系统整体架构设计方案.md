## 1. 系统概述

### 1.1 产品定位

**OpenClaw** 是一个**个人 AI 助手系统**，用户可在自有设备上运行。它通过统一的消息网关连接多个通讯渠道，提供智能化的对话体验和可扩展的工具能力。

### 1.2 核心特性

| 特性 | 说明 |
|------|------|
| **多通道消息网关** | 支持 WhatsApp、Telegram、Slack、Discord、Signal、iMessage、Google Chat 等 12+ 通道 |
| **AI Agent 运行时** | 基于 Pi RPC 的 Agent 运行时，支持多模型（Claude、GPT、Gemini 等） |
| **插件扩展系统** | 支持 Channel、Tool、Gateway RPC、CLI Command、Hook 等多种扩展 |
| **跨平台客户端** | CLI、WebChat、macOS 菜单栏应用、iOS/Android 节点 |
| **本地优先架构** | 数据存储在本地，支持远程访问（Tailscale/SSH） |
| **安全沙箱** | 支持 Docker 沙箱隔离非主会话的执行 |

### 1.3 技术栈

```yaml
语言: TypeScript (ESM)
运行时: Node.js ≥22
包管理: pnpm (主) / bun (可选)
Agent 运行时: @mariozechner/pi-agent-core
协议定义: TypeBox + JSON Schema
测试框架: Vitest + V8 Coverage
代码规范: Oxlint + Oxfmt
构建工具: tsc + tsx
```

## 7. 安全设计

### 7.1 安全模型

```
┌─────────────────────────────────────────────────────────────────┐
│                      Security Layers                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Layer 1: Network Security                                       │
│  ├── Bind address (127.0.0.1 / loopback / all)                   │
│  ├── Auth token (optional but recommended)                       │
│  └── Tailscale/VPN for remote access                             │
│                                                                  │
│  Layer 2: Device Trust                                           │
│  ├── Device pairing (challenge/response)                         │
│  ├── Device token persistence                                    │
│  └── Local auto-approval (same-host)                             │
│                                                                  │
│  Layer 3: Message Security                                         │
│  ├── DM Pairing (unknown senders)                                │
│  ├── Allowlist (channels.allowFrom)                              │
│  └── Group mention gating                                        │
│                                                                  │
│  Layer 4: Execution Security                                       │
│  ├── Tool allowlist/denylist                                     │
│  ├── Execution approval (elevated)                               │
│  └── Docker sandbox (non-main sessions)                          │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 7.2 安全策略

| 场景 | 策略 | 配置 |
|------|------|------|
| 未知发送者 DM | 发送配对码，不处理消息 | `dmPolicy: "pairing"` |
| 群组消息 | 需要 @提及 或回复 | `requireMention: true` |
| 工具执行 | 主会话直接执行，其他会话沙箱 | `sandbox.mode: "non-main"` |
| 危险工具 | 需要审批 | `approval: true` |

## 10. 关键技术决策

### 10.1 为什么选择 Pi RPC？

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| **Pi RPC** | 流式工具执行、状态管理、成熟的 Agent 运行时 | 依赖外部库 | ✅ |
| 直接 API 调用 | 简单直接 | 无流式、无状态管理 | ❌ |
| LangChain | 生态丰富 | 过于复杂、流式支持差 | ❌ |

### 10.2 为什么选择 WebSocket 作为控制平面？

| 特性 | HTTP | WebSocket | gRPC |
|------|------|-----------|------|
| 实时双向 | ❌ | ✅ | ✅ |
| 浏览器兼容 | ✅ | ✅ | ❌ |
| 流式支持 | 复杂 | 原生 | 原生 |
| 调试便利 | ✅ | ✅ | ❌ |
| 选择 | - | ✅ | ❌ |

### 10.3 插件加载为什么选择 jiti？

- 运行时加载 TypeScript 无需预编译
- 与 Node.js ESM 兼容
- 支持 source map
- 轻量且成熟

## 11. 性能考量

### 11.1 设计目标

| 指标 | 目标 | 策略 |
|------|------|------|
| 消息延迟 | < 100ms (无 AI) | 异步处理，无阻塞 |
| 并发连接 | 1000+ | WS 连接池，事件驱动 |
| AI 首字延迟 | < 2s | 模型缓存，连接复用 |
| 内存占用 | < 500MB | 流式处理，及时释放 |

### 11.2 优化策略

1. **流式处理**: AI 响应流式传输，减少等待
2. **连接复用**: 通道连接长期保持
3. **异步非阻塞**: 所有 I/O 操作异步
4. **增量更新**: 配置热重载，无需重启
